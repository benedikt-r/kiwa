Dynamic Rails Web App

4.5: Views

Learning Goals

  • Add featured products dynamically in an image carousel on the landing page by writing embedded Ruby to loop and display an array of stored data
  • Create and render a product partial on the main Products page using correct partial path, and describe the purpose of the partial

 Estimated Read Time: 25 Minutes.

Introduction

Hello, again! Now that you've learned about the M and the C of MVC (models and controllers), well focus on the V (views). In a Rails app, when a browser makes a request to the app, this request gets routed from the routes file to an action within the appropriate controller file, and from there (if it's a GET request) to the view that corresponds to that action. Let's take a closer look, by starting with the layouts.

Layouts

Before the specific view gets loaded, the controller loads a layout file. Layouts are the main templates that wrap around your views, and they are found in the app/views/layouts folder.

The default layout in a Rails app is the application.html.erb file, and it contains your header and footer and possibly some other content (like a navigation bar) that will be loaded for every page on your site. You can also create additional layout files and specify them by including the render method in your controller actions. For instance, if you wanted to override the default layout file application.html.erb and call a different layout file named products.html.erb, you would do it like this:

def index
  @products = Product.all
  render layout: "products"
end

You can also specify the layout at the top of your controller and it will apply that layout to all of the actions within that controller:

class SimplePagesController < ApplicationController
  layout "static"
...

Yield
If you open up your application.html.erb layout, you will see a single line in the body of the page that says: <%= yield %>. The yield tells the application where to render the view files inside the layout. So if your browser requests the index action of a controller, the index view file will be loaded here, inside the body tags of the application layout.

Displaying Dynamic Data in an Image Carousel

Now that you've probably added a handful of test products to your database and linked to Bootstrap in your header, lets add an image carousel feature on the homepage that will be populated with dynamic content. On the backend, we will be pulling the first three products from the database and displaying them as the individual slides, with all the content (image, name, description, link) populated dynamically. So if our products ever get updated, the new ones will appear on the homepage without needing to modify any code. To see how this will be achieved, we'll use an example from your products controller. This is the code that was generated by the Rails scaffolding:

def index
  @products = Product.all
end

This takes all of your products from the database and populates them into an array called @products. To limit this to the first three items, we can replace "all" with just a "limit". The query remains the same, but only three records, at most, are retrieved.

def index
  @products = Product.limit(3)
end

This is what we want to replicate, so insert that query into your landing_page action in the simple pages controller:

def landing_page
  @products = Product.limit(3)
end

Now let's look at what happens on the frontend. We'll use the products Index page again as an example. Open up the products/index view and find the block that starts with this line:

<% @products.each do |product| %>

This block takes the variable that was set in the controller and iterates through each record (@products.each). Then each individual product is set to a variable called product (do |product|), and then within the loop, the attributes of the product are displayed (<%= product.name %>). We will essentially be doing the same thing on the landing page, but instead of populating a table we will be displaying the products in the carousel.

If you look at the carousel example code, the blocks that get repeated are called <div class="item"> (each slide is contained in one of these item divs). So this will go inside our @products.each block. For the image tag, you can insert this line of code to dynamically generate an image tag for your product:

<%= image_tag(product.image_url) %>

And for the caption, put the product name:

<div class="carousel-caption d-none d-md-block">
  <%= product.name %>
</div>

The condensed version of the carousel slides block will then look like this:

<div class="carousel-inner">
  <% @products.each do |product| %>
  <div class="carousel-item">
    <%= image_tag(product.image_url) %>

    <div class="carousel-caption d-none d-md-block">
      <%= product.name %>
    </div>
  </div>
  <% end %>
</div>

As long as you have names and product URLs for all of your products, these fields should populate. Take a look at the HTML that gets generated once you load the page to check if these fields are populating correctly.

According to the Bootstrap documentation, your carousel won't be visible unless one of the slides also has the class active. This determines which slide appears visible when the page loads. So let's set the active slide to the first item in the array. We can add a conditional in one concise line of ERB code (this replaces the <div class="item"> line):

<div class="carousel-item<%= " active" if product == @products.first %>">

So when the HTML gets generated, your source code will appear populated, with the first item having the class active:

<div class="carousel-inner">
  <div class="carousel-item active">
    <img alt="Red city bike" src="http://myapp.com/img/red_city_bike.jpg" />
    <div class="carousel-caption d-none d-md-block">
      <p>Red City Bike</p>
    </div>
  </div>

</div>

Tidying Up with Partials

Take a look at the form inside your app/views/products folder. Open up your new and edit views; they both contain this line:

<%= render 'form' %>

The form being referenced here is a type of view called a partial. Its short for partial template, meaning its a part of a view that is typically shared across several views. Its another way of using DRY principles to reduce the amount of code that gets duplicated, but it can also be used to consolidate a long view into smaller pieces. In this example, the form in the new and edit templates contains the same fields, so theres no need for us to write the form twice.

Notice the file name starts with an underscore (_). Partials are always named using an underscore at the beginning of the filename. To call a partial from within a view that's in the same folder as the view, use the ERB tag with the equals sign, calling render partial: and the name of the partial, without the underscore and in quotes (single or double quotes are both fine).

<%= render partial: 'form' %>

You can remove the "partial:" part if you arent passing in any other parameters besides the file name, as in your product views. You can also render a partial that exists in a separate folder by including the folder name:

<%= render 'shared/form' %>

It's common to use a shared folder (in app/views/shared) to store partials that may be used across several different controllers.

Scope

Remember that instance variables (variables that start with @) can be accessed by any views loaded by the controller where the variable is set. This also applies to partialsyou can access your @products variable from any partials called inside the index.html.erb file. However, local variables need to be passed in manually for them to be displayed inside the partial. In this case, we need to use the longer form, specifying each parameter, so we have to call render partial: first:

<%= render partial: 'shared/form', locals: { product: product } %>

This will enable you to access the product variable inside the partial.

Lets try an example on our products Index page. We can take the block of code inside the each loop. (Keep in mind, that most likely this block looks differently for you depending on how you decided to design your products Index page!):

<div class="row">
  <% @products.each do |product| %>
    <div class="col-sm">
      <h1><%= link_to product.name, product %></h1>
      <p><%= product.description %></p>
      <strong>$ <%= product.price %></strong>
    </div><!-- /col -->
  <% end %>
</div>

Put everything within the @products.each do ... end block inside a partial called _product.html.erb, which will display each individual products information. Now you can condense the table in the index view, like so:

...
  <% @products.each do |product| %>
    <%= render partial: "product", locals: {product: product} %>
  <% end %>

But there is also a shorthand method available in this case. If you have a partial that's the same name of a model (in this case, product), it will automatically set a local variable of the same name. Since that's also the variable we are passing to it, we can just call it like this:

<% @products.each do |product| %>
  <%= render product %>
<% end %>

But wait! Theres more! Because @products is an array of objects of the same model (called a collection in Rails terminology), we can condense this entire block into just one line where we pass the array directly into the partial using the collection option:

<%= render partial: "product", collection: @products %>

Even better, in cases like this where the objects of the collection (Product objects in this case) match the name of the partial, you can shorten this down even further:

<%= render @products %>

We dont even have to alter the code inside the partial for any of the above examples to work. Rails is loaded with tricks like this that will allow you to condense your code into smaller blocks... and we like simplifying our code!

Summary

You'll encounter collections very often in Rails (any time you need to list data, essentially), so we recommend you spend some time reading through the Rails Guide linked in the Learn More section to familiarize yourself.
